<task type="auto">
  <name>Task 1: Create Amplify build specification</name>
  <files>
    amplify.yml
  </files>
  <action>
    Create `amplify.yml` in the project root. This tells Amplify Hosting how to build and where to find output.

    The app uses:
    - pnpm (not npm)
    - Nx workspace with `nx build` producing output in `dist/word-list`
    - Angular 21 SPA that needs all routes to fall back to index.html

    ```yaml
    version: 1
    frontend:
      phases:
        preBuild:
          commands:
            - corepack enable
            - pnpm install --frozen-lockfile
        build:
          commands:
            - pnpm build
      artifacts:
        baseDirectory: dist/word-list/browser
        files:
          - '**/*'
      cache:
        paths:
          - node_modules/**/*
    ```

    Note: Angular 21 with esbuild outputs to `dist/word-list/browser` (the `/browser` subdirectory is the esbuild output convention). Verify this by checking the actual build output directory.

    Also create a `customHttp.yml` or check if Amplify needs SPA redirect config. For Angular SPAs, all routes need to return index.html. Amplify Hosting handles this via a redirect rule that can be set via CLI or console. The CLI approach:

    After the app is created, add a rewrite rule: `[{"source": "/<*>", "target": "/index.html", "status": "200"}]`
  </action>
  <verify>
    amplify.yml exists in project root.
    pnpm build produces output in dist/word-list/browser/ (verify path is correct).
  </verify>
  <done>Build spec created and verified against actual build output path.</done>
</task>

<task type="auto">
  <name>Task 2: Deploy to Amplify Hosting via CLI</name>
  <files>
    (no source files — CLI operations only)
  </files>
  <action>
    Use the Amplify CLI and AWS CLI to connect the repo and deploy.

    Steps:
    1. Check if the Amplify app already exists: `aws amplify list-apps` — the backend is already deployed, so there should be an existing Amplify app.
    2. If app exists, get its app ID. If not, the Amplify Gen 2 deployment may have created it during `npx ampx sandbox` or `npx ampx deploy`.
    3. Connect the repo branch for hosting. Amplify Gen 2 apps may already have hosting configured — check first.
    4. If hosting needs to be set up: use `aws amplify create-branch --app-id <id> --branch-name main` and configure auto-build.
    5. Add the SPA rewrite rule: `aws amplify update-app --app-id <id> --custom-rules '[{"source":"/<*>","target":"/index.html","status":"200"}]'`
    6. Trigger a build: `aws amplify start-job --app-id <id> --branch-name main --job-type RELEASE`
    7. Wait for build to complete and get the default URL.

    If the Amplify Gen 2 setup uses a different approach (e.g., `ampx` CLI for hosting), use that instead. Read the Amplify Gen 2 docs for the correct hosting setup method.

    The key requirement: the frontend must be built and served from Amplify Hosting, hitting the already-deployed Amplify backend.
  </action>
  <verify>
    aws amplify get-app --app-id <id> shows the app with hosting configured.
    The default URL (e.g., main.d1234abcdef.amplifyapp.com) is accessible.
  </verify>
  <done>App deployed to Amplify Hosting. Default URL is live.</done>
</task>
