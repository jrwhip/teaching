  <files>
    src/app/practice/math/shared/math-results.service.ts

  <action>
    The current MathResultsService has methods like `loadStudentStats()`, `loadPerformanceCounters()`, `loadRecentAttempts()` that are async methods awaiting Amplify calls and stuffing results into signals.

    Rewrite read operations using rxResource:
    ```typescript
    // Student stats loaded reactively based on authenticated user
    readonly studentStats = rxResource({
      request: () => this.auth.userProfile.value()?.id,  // triggers when user changes
      stream: ({ request: studentId }) => {
        if (!studentId) return of(null);
        return defer(() => this.data.models.PerformanceCounter
          .listPerformanceCounterByStudentId({ studentId }))
          .pipe(
            map(({ data }) => this.aggregateStats(data)),
            catchError(() => of(null))
          );
      }
    });
    ```

    For recent attempts:
    ```typescript
    readonly recentAttempts = rxResource({
      request: () => this.auth.userProfile.value()?.id,
      stream: ({ request: studentId }) => {
        if (!studentId) return of([]);
        return defer(() => this.data.models.ProblemAttempt
          .listProblemAttemptByStudentIdAndAttemptedAt(
            { studentId },
            { sortDirection: 'DESC', limit: 20 }
          ))
          .pipe(
            map(({ data }) => data ?? []),
            catchError(() => of([]))
          );
      }
    });
    ```

    Key rules:
    - rxResource handles loading state automatically (.isLoading())
    - rxResource handles error state
    - The `request` function makes resources reactive — when auth state changes, data reloads
    - All Amplify calls wrapped in `defer()`, NEVER `from()`
    - Remove the manual signal state (sessionStats, recentAttempts signals that were being manually .set())
    - Remove all `async` methods for reads
    - Remove `loadStudentStats()`, `loadPerformanceCounters()`, `loadRecentAttempts()` — rxResource replaces them
    - Keep `startNewSession()` if it just resets local state — that's fine as a synchronous method
  </action>

<task type="auto">
  <name>Task 2: Rewrite MathResultsService mutation operations with defer()</name>
  <files>
    src/app/practice/math/shared/math-results.service.ts
  </files>
  <action>
    The `recordAttempt()` method is currently async — it awaits Amplify create calls for ProblemAttempt and PerformanceCounter updates.

    Rewrite to return Observable:
    ```typescript
    recordAttempt(attempt: AttemptInput): Observable<void> {
      const profile = this.auth.userProfile.value();
      if (!profile) return EMPTY;  // not authenticated, skip

      return defer(() => this.data.models.ProblemAttempt.create({
        studentId: profile.id,
        ...attempt,
        attemptedAt: new Date().toISOString(),
      })).pipe(
        switchMap(() => this.updatePerformanceCounter(attempt, profile.id)),
        tap(() => {
          // Update local session state (signals for current streak, session score)
          this.updateSessionState(attempt);
          // Trigger rxResource reload for stats
          this.studentStats.reload();
        }),
        catchError(err => {
          console.error('Failed to record attempt:', err);
          return EMPTY;  // Don't break the quiz flow on recording failure
        })
      );
    }
    ```

    The components that call `recordAttempt()` currently fire-and-forget (they don't await the result). Make them subscribe:
    ```typescript
    this.results.recordAttempt(attempt).subscribe();
    ```

    Or if the component doesn't care about the result at all, keep fire-and-forget via `.subscribe()`.

    For `updatePerformanceCounter()` — same pattern, wrap in defer(), return Observable.

    Key rules:
    - Mutations return Observable<void>
    - Caller subscribes (one-shot, no cleanup needed for Amplify calls)
    - Error handling in the service, not the component
    - Local session state (streaks, score) updated synchronously via tap()
    - rxResource .reload() triggers fresh data fetch after mutation
  </action>
