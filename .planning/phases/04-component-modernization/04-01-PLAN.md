  <action>
    **DashboardRedirectComponent:**
    Currently uses `ngOnInit` to read the user profile and navigate based on role. This should be a route guard or resolver, not a component with ngOnInit.

    Replace with an `effect()` that reacts to the auth profile signal:
    ```typescript
    constructor() {
      effect(() => {
        const profile = this.auth.userProfile.value();
        if (profile) {
          const route = profile.role === 'TEACHER' ? '/dashboard/teacher'
            : profile.role === 'PARENT' ? '/dashboard/parent'
            : '/dashboard/student';
          this.router.navigate([route]);
        }
      });
    }
    ```
    Or better: convert to a functional route guard that handles the redirect before the component loads. That would be the cleanest approach — no component needed at all.

    Preferred approach: Replace with a route guard function in the dashboard routes:
    ```typescript
    // dashboard.routes.ts
    export const dashboardRedirectGuard: CanActivateFn = () => {
      const auth = inject(AuthService);
      const router = inject(Router);
      const profile = auth.userProfile.value();
      if (!profile) return router.createUrlTree(['/login']);
      const route = profile.role === 'TEACHER' ? '/dashboard/teacher' : ...;
      return router.createUrlTree([route]);
    };
    ```
    Then remove DashboardRedirectComponent entirely.

    **MathMenuComponent:**
    Currently uses `ngOnInit` to call `this.results.loadStudentStats()`. After Phase 3, MathResultsService provides rxResource-based signals. The component should just read those signals directly.

    Remove `implements OnInit` and the `ngOnInit()` method. The template should read:
    - `this.results.studentStats.value()` for stats
    - `this.results.studentStats.isLoading()` for loading state
    - `this.results.recentAttempts.value()` for recent attempts

    No initialization needed — rxResource loads automatically when the signal dependencies change.
  </action>

<task type="auto">
  <name>Task 2: Fix TeacherResultsComponent and ParentResultsComponent</name>
  <files>
    src/app/practice/math/results/teacher-results.component.ts,
    src/app/practice/math/results/parent-results.component.ts
  </files>
  <action>
    Both components have massive `async ngOnInit()` methods that waterfall through Amplify calls (classrooms → enrollments → students → counters). This is the most complex data loading in the app.

    Replace with rxResource pattern. These components need their own resources since they query across multiple students.

    For TeacherResultsComponent:
    ```typescript
    private readonly auth = inject(AuthService);
    private readonly data = inject(DataService);

    readonly studentsResource = rxResource({
      request: () => this.auth.userProfile.value(),
      stream: ({ request: profile }) => {
        if (!profile || profile.role !== 'TEACHER') return of([]);
        return defer(() => this.data.models.Classroom
          .listClassroomByTeacherId({ teacherId: profile.id }))
          .pipe(
            switchMap(({ data: classrooms }) =>
              this.loadStudentSummaries(classrooms ?? [])
            ),
            catchError(() => of([]))
          );
      }
    });

    // Loading/error come from resource
    readonly loading = computed(() => this.studentsResource.isLoading());
    readonly students = computed(() => this.studentsResource.value() ?? []);
    ```

    Move the student selection to a signal + rxResource:
    ```typescript
    readonly selectedStudentId = signal<string | null>(null);

    readonly selectedAttemptsResource = rxResource({
      request: () => this.selectedStudentId(),
      stream: ({ request: studentId }) => {
        if (!studentId) return of([]);
        return defer(() => this.data.models.ProblemAttempt
          .listProblemAttemptByStudentIdAndAttemptedAt(
            { studentId },
            { sortDirection: 'DESC', limit: 20 }
          )).pipe(
            map(({ data }) => data?.map(a => ({ ...mapped fields... })) ?? []),
            catchError(() => of([]))
          );
      }
    });
    ```

    For the waterfall of classrooms → enrollments → studentIds → counters, use a helper method that chains defer() calls with switchMap/mergeMap:
    ```typescript
    private loadStudentSummaries(classrooms: Classroom[]): Observable<StudentSummary[]> {
      return forkJoin(
        classrooms.map(c =>
          defer(() => this.data.models.ClassroomEnrollment
            .listClassroomEnrollmentByClassroomId({ classroomId: c.id }))
        )
      ).pipe(
        map(results => results.flatMap(r => r.data ?? []).filter(e => e.isActive)),
        switchMap(enrollments => {
          const studentIds = [...new Set(enrollments.map(e => e.studentId))];
          return forkJoin(studentIds.map(id => this.loadStudentSummary(id)));
        })
      );
    }
    ```

    Use `forkJoin` for parallel fetches (replacing the sequential for...of loops that were converted to Promise.all in Phase 2).

    Apply the exact same pattern to ParentResultsComponent (it has the same structure but queries ParentStudentLink instead of Classroom).

    Remove `implements OnInit` and `ngOnInit()` from both components. Remove all `async` methods.
  </action>
