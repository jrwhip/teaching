<task type="auto">
  <name>Task 1: Convert auth components to reactive forms</name>
  <files>
    src/app/auth/login/login.component.ts,
    src/app/auth/register/register.component.ts,
    src/app/auth/confirm/confirm.component.ts,
    src/app/auth/forgot-password/forgot-password.component.ts
  </files>
  <action>
    Each auth component currently uses FormsModule with [(ngModel)] bindings. Convert to reactive forms:

    1. Replace `FormsModule` import with `ReactiveFormsModule` in the component's `imports` array
    2. Create a typed FormGroup using `FormControl` for each field
    3. Replace `[(ngModel)]="field"` with `[formControl]="form.controls.field"` or `formControlName="field"` in templates
    4. Replace the signal-based state for form fields with FormGroup/FormControl
    5. Keep the existing signal-based state for non-form state (loading, error messages)

    Pattern for login:
    ```typescript
    private readonly fb = inject(NonNullableFormBuilder);
    readonly form = this.fb.group({
      email: ['', [Validators.required, Validators.email]],
      password: ['', Validators.required],
    });
    ```

    For the templates (inline templates):
    - Replace `[(ngModel)]="email()"` → `formControlName="email"`
    - Wrap inputs in `<form [formGroup]="form">`
    - Access values via `this.form.getRawValue()` in submit methods

    These are inline template components, so both the template and the class are in the same .ts file.

    Do NOT change the auth logic itself (signIn, signUp, confirmSignUp, resetPassword calls). Only change how form data is collected.
  </action>
  <verify>
    grep -rn "FormsModule" src/app/auth --include="*.ts" returns zero results.
    grep -rn "ngModel" src/app/auth --include="*.ts" returns zero results.
    pnpm build passes.
  </verify>
  <done>All 4 auth components use ReactiveFormsModule with typed FormGroups. Zero ngModel bindings. Build passes.</done>

    The math practice components use FormsModule for simple text inputs (user answer input). For these simple single-input components where the input is already managed via signals (userInput signal), the FormsModule usage is typically just for `[(ngModel)]` on the answer input field.

    For each math component:
    - If it only uses `[(ngModel)]` on a single text input for answer entry, convert to a reactive FormControl
    - Replace the signal-based userInput with a FormControl, or use `(input)` event binding to update the existing signal (simpler refactor for components that will be further modernized in Phase 4)

    For the legacy math.component.ts: This will be fully rewritten in Phase 4 (plan 04-02). For now, just replace FormsModule with ReactiveFormsModule and add a FormControl for the answer input.

    For quiz/quiz2/quiz3 and number-line/area-perimeter: These have signal-based state that works well. The simplest compliant approach is:
    - Keep the signal for state
    - Use `(input)` event binding instead of [(ngModel)]: `<input (input)="userInput.set($any($event.target).value)">`
    - Wait — that uses $any. Better: use a typed event handler or FormControl.
    - Best approach: Replace FormsModule with ReactiveFormsModule, create a FormControl, use `[formControl]` binding, and sync with existing signals via `this.answerControl.valueChanges` if needed.

    The goal is zero FormsModule imports. ReactiveFormsModule is acceptable, or no forms module at all if using plain event bindings with signals.
